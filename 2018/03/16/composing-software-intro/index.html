합성 : 전체를 구성하기 위해 부분이나 요소를 결합하는 행위

고등학교 프로그래밍 수업 첫 시간에 소프트웨어 개발이란 복잡한 문제를 작은 문제들로 분해하고 작은 문제들의 해법들을 다시 조합해 복잡한 문제를 해결 할 수 있는 솔루션을 만드는 행위라고 배웠습니다.

제 인생에서 가장 후회되는 것 중 하나는 그 교훈의 중요성을 일찍이 깨닫지 못 했던 것입니다. 전 너무 늦게 소프트웨어 설계의 본질을 깨달았습니다.

수백 명의 개발자들을 인터뷰하며 나만 그런 것이 아님을 알게됐습니다. 아주 소수의 개발자만이 소프트웨어 개발의 본질에 대해 잘 알고 있었습니다. 대부분의 경우에는 우리가 활용 할 수있는 가장 중요한 개념과 그것을 잘 적용하는 방법을 몰랐습니다. 다음은 소프트웨어 개발에서 가장 중요한 두가지 질문입니다.

  • 함수 합성 function composition 이란 무엇인가?
  • 객체 합성 object composition이란 무엇인가?

문제는 당신이 단지 잘 모른다고해서 객체 및 함수 합성을 사용하지 않을 수는 없다는 것입니다. 당신은 여전히 합성을 합니다 -하지만 나쁘게합니다. 그렇기에 버그가 넘쳐나고 다른 개발자가 이해하기 어려운 코드를 작성하게 됩니다. 이는 큰 문제이며 적지않은 대가를 지불하고 있습니다. 우리는 소프트웨어를 만드는 것보다 소프트웨어를 유지 보수하는 데 더 많은 시간을 투자하며, 버그는 전 세계 수십억 명의 사람들에게 영향을줍니다.

오늘날 전 세계 어디에든 소프트웨어가 있습니다. 모든 자동차는 바퀴가 달린 미니 수퍼 컴퓨터이며 소프트웨어 설계상의 문제점으로 인해 실제 사고가 발생하고 사람이 목숨을 잃습니다. 2013년, 배심원단은 사고 조사하는 과정에서 Toyota의 소프트웨어 개발팀이 10,000 개의 글로벌 변수가있는 스파게티 코드를 작성한 것을 발견했습니다.

해커 및 정부는 버그를 활용해 사람들을 감시하고, 신용 카드를 훔치고, DDoS 공격을 하고 , 암호를 해독하고, 심지어 선거를 조작 합니다.

우리가 더 잘해야합니다.

사실 당신은 매일 소프트웨어를 합성합니다

여러분이 소프트웨어 개발자라면 의식하든 하지 못하든 매일매일 함수와 데이터 구조를 합성합니다. 조금만 신경을 쓰면 더 잘 할 수 있지만, 그렇지 않고 덕트 테이프와 미친 접착제crazy glue로 모든걸 망쳐버릴 수도 있습니다.

소프트웨어 개발 프로세스란 큰 문제를 작은 문제로 분해하여 작은 문제를 해결하는 요소를 만들고 이러한 구성 요소들을 합성하여 완전한 응용 프로그램을 구성하는 것입니다.

함수 합성

함수 합성은 다른 함수의 출력에 함수를 적용하는 과정입니다. 대수학에서 f 와 g 두 함수, 그리고 합성 함수 (f ∘ g)(x) = f(g(x)) 를 본적 있을 겁니다. 조그마한 원 기호는 합성 연산자입니다. 일반적으로 "composed with"또는 "after"로 발음[1]됩니다. f 와 g를 합성하는 것은 f에 g에 x 와 같이 부를 수 있습니다 . 이 때 g 가 먼저 평가되기 때문에 g 를 계산하고 그 결과를 f의 인수로 전달합니다.

pipe() 유틸리티는 한 함수의 출력을 다른 함수의 입력으로 전달하여 함수 파이프 라인을 생성합니다. pipe() (그리고 쌍둥이라고 할 수 있는 compose() )를 사용하면 중간 변수intermediary variables가 필요 없습니다. 인수를 언급하지 않고 함수를 작성하는 것을 포인트없는 스타일point-free style 혹은 무인수 방식이라고 합니다. 이를 위해 함수를 명시적으로 선언하는 대신 다른 함수를 반환하는 함수를 호출합니다. 즉, function 키워드 나 화살표 구문 ( => )이 필요하지 않습니다.

무인수 방식으로 전체 로직을 구성할 수도 있지만, 중간 변수들이 여러분의 기능에 불필요한 복잡성complexity을 더하기 때문에 여기 저기 조금씩 사용하는 것도 좋습니다.

복잡성 감소에는 몇 가지 이점이 있습니다.

작업 메모리

인간의 두뇌는 작업 메모리 에서 이산 양자 (discrete quanta)를위한 공유 자원을 거의 가지고 있지 않으며, 각 변수는 잠재적으로 그 양자 중 하나를 소비합니다. 변수를 더 추가하면 각 변수의 의미를 정확히 불러 낼 수있는 능력이 떨어집니다. 작업 메모리 모델은 일반적으로 4-7 개의 이산 퀀텀을 포함합니다.이 수치보다 크면 오류율이 급격히 증가합니다.

파이프 형식을 사용하여 다른 변수에 사용할 수있는 작업 메모리의 거의 절반을 확보하여 3 가지 변수를 제거했습니다. 그것은인지 부하를 상당히 줄입니다. 소프트웨어 개발자는 평균적인 사람보다 작업 메모리에 데이터를 청크하는 경향이 있지만 보존의 중요성을 약화시키는 것은 아닙니다.

신호 대 잡음비

간결한 코드는 또한 코드의 신호 대 잡음 비율을 향상시킵니다. 라디오를 청취하는 것과 같습니다. 라디오가 방송국에 제대로 맞춰져 있지 않으면 많은 소음이 발생하며 음악을 듣기가 더 어려워집니다. 정확한 방송국으로 튜닝하면 소음이 사라지고 음악 신호가 강해집니다.

코드는 같은 방법입니다. 보다 간결한 코드 표현은 이해력을 향상시킵니다.일부 코드는 유용한 정보를 제공하고 일부 코드는 공간을 차지합니다. 전송되는 의미를 줄이지 않고 사용하는 코드의 양을 줄이면 코드를 읽고 이해해야하는 다른 사람들이 코드를 더 쉽게 이해할 수 있습니다.

벌레의 표면적

전후 기능을 살펴보십시오. 기능이 다이어트를하고 무게가 많이 나가는 것처럼 보입니다. 이것은 여분의 코드가 버그를 숨길 수있는 추가 표면적을 의미하기 때문에 중요합니다. 즉 더 많은 버그가 숨어 버릴 수 있음을 의미합니다.

적은 코드 = 버그가 적은 표면 = 버그가 적습니다.

"컴퓨터 과학에서 복합 데이터 유형 또는 복합 데이터 유형은 프로그래밍 언어의 원시 데이터 유형 및 기타 복합 유형을 사용하여 프로그램에서 구성 할 수있는 모든 데이터 유형입니다. […] 합성 유형을 구성하는 행위는 구성으로 알려져 있습니다. "~ Wikipedia

다음은 프리미티브입니다.

그리고 이것은 복합체입니다.

마찬가지로 모든 배열, 집합, 맵, 약식 맵, 형식화 된 배열 등은 복합 데이터 유형입니다. 비 기본 데이터 구조를 작성할 때마다 어떤 종류의 오브젝트 구성을 수행합니다.

Gang of Four는 개별 구성 요소와 집계 된 복합 요소를 동일하게 처리 할 수있는 재귀 개체 구성의 특정 유형 인 복합 패턴이라는 패턴을 정의합니다. 일부 개발자는 복합 패턴이 오브젝트 합성의 유일한 형태 라고 생각하면서 혼란스러워 합니다. 혼동하지 마십시오. 객체 구성에는 여러 가지 종류가 있습니다.

Gang of Four는 계속해서 "디자인 패턴에 반복해서 적용된 오브젝트 구성을 보게 될 것입니다"라고 말한 다음 위임 (state, strategy 및 visitor 패턴에서 사용 된), 지인 (객체가 참조로 다른 객체를 알고있을 때 일반적으로 매개 변수로 전달됨 : 네트워크 요청 처리기가 로거에 대한 참조를 전달하여 요청을 기록 - 요청 처리기 가 로거를 사용함 ) 및 집계 (자식 객체가 부모 객체의 일부를 형성하는 경우 : a - a 관계, 예를 들어 DOM 하위는 DOM 노드의 구성 요소 - DOM 노드 는 하위를 가짐).

클래스 상속은 복합 객체를 구성하는 데 사용할 수 있지만 제한적이고 취약한 방법입니다. Gang of Four는 "클래스 상속에 대한 객체 구성 선호"라고 말하면서 클래스 상속의 강고하고 단단히 결합 된 접근 방식보다는 객체를 복합하는 데 유연한 접근 방식을 사용하도록 조언합니다.

“컴퓨터 과학의 범주 형 방법 : 토폴로지의 측면에서” (1989)의 개체 구성에 대한보다 일반적인 정의를 사용합니다.

“복합 물체는 물체들을 합쳐서 만들어 지는데, 각각의 물체는 이전 물의 일부분이된다.”

또 다른 좋은 참고 자료는 “Composite Design을 통한 신뢰할 수있는 소프트웨어”(Glenford J Myers, 1975)입니다. 두 책 모두 절판되었지만 Amazon에서의 판매 인이나 이베이에서 여전히 책을 찾을 수 있습니다. 기술적 인 깊이.

클래스 상속은 복합 객체 생성의 한 종류에 불과합니다. 모든 클래스는 복합 객체를 생성하지만 모든 복합 객체가 클래스 또는 클래스 상속에 의해 생성되는 것은 아닙니다. "클래스 상속에 대한 객체 합성 우선"은 클래스 계층의 조상 (ancestor)에서 모든 속성을 상속하지 않고 작은 구성 요소 부분에서 복합 객체를 형성해야한다는 것을 의미합니다. 후자는 객체 지향 설계에서 다양한 잘 알려진 문제를 일으 킵니다.

  • 밀 결합 문제 : 자식 클래스는 부모 클래스의 구현에 의존하기 때문에 클래스 상속은 객체 지향 디자인에서 사용할 수있는 가장 조밀 한 결합입니다.
  • 깨지기 쉬운 기본 클래스 문제 : 긴밀한 결합으로 인해 기본 클래스가 변경되면 잠재적으로 제 3자가 관리하는 코드에서 많은 수의 클래스가 손상 될 수 있습니다. 작성자는 알지 못하는 코드를 깨뜨릴 수 있습니다.
  • 유연하지 못한 계층 구조 문제 : 단일 조상 분류법으로 충분한 시간과 진화가 이루어지면 모든 클래스 분류가 새로운 사용 사례에 대해 결국 잘못된 것입니다.
  • 필요성에 의한 중복 문제 : 유연하지 않은 계층 구조로 인해 새로운 유스 케이스가 종종 확장이 아닌 복제에 의해 구현되어 예기치 않게 분기되는 유사한 클래스로 이어집니다. 복제가 설정되면 어떤 클래스의 새 클래스가 내려야하는지 또는 그 이유가 명확하지 않습니다.

결론

이것은 FP (Functional Programming)와 OOP (Object-Oriented Programming) 또는 다른 언어에 대한 것이 아닙니다. 구성 요소는 함수, 데이터 구조, 클래스 등의 형태를 취할 수 있습니다. 다른 프로그래밍 언어는 구성 요소에 대해 서로 다른 원자 요소를 제공하는 경향이 있습니다. 자바는 클래스를 제공하고, 하스켈은 함수를 제공한다. 그러나 어떤 언어와 어떤 패러다임을 선호하든 관계없이 함수와 데이터 구조를 구성 할 수 없다. 결국, 그것이 모두 뒤죽박죽이다.

함수 프로그래밍은 JavaScript로 작성하는 것이 가장 간단하기 때문에 함수 프로그래밍 커뮤니티는 함수 작성 기술을 공식화하는 데 많은 시간과 노력을 투자했습니다.

우리가하지 못하는 것은 함수형 프로그래밍이 객체 지향 프로그래밍보다 낫다는 것입니다. 그렇지 않으면 하나를 선택해야합니다. OOP 대 FP는 잘못된 이분법입니다. 최근 몇 년 동안 본 모든 실제 Javascript 애플리케이션은 FP와 OOP를 광범위하게 혼합합니다.

객체 합성을 사용하여 함수 프로그래밍을위한 데이터 유형을 생성하고 OOP를위한 객체를 생성하는 함수 프로그래밍을 생성합니다.

소프트웨어를 작성하는 방법에 상관없이 잘 작성해야합니다.

소프트웨어 개발의 핵심은 구성입니다.

구성을 이해하지 못하는 소프트웨어 개발자는 볼트 나 못에 대해 모르는 주택 건설업자와 같습니다. 구성을 의식하지 않고 소프트웨어를 제작하는 것은 집 빌더가 덕트 테이프와 미친 접착제로 벽을 붙이는 것과 같습니다.

이제는 단순화 할 때가되었으며, 단순화하는 가장 좋은 방법은 본질에 도달하는 것입니다. 문제는, 업계의 거의 아무도 본질에 대한 좋은 핸들을 가지고 있지 않다는 것입니다. 우리 업계는 소프트웨어 개발자 인 당신에게 실패했습니다. 개발자를 더 잘 훈련시키는 것은 업계의 책임입니다. 우리는 향상시켜야합니다. 우리는 책임을 져야합니다. 모든 것이 경제에서 의료 장비에 이르기까지 오늘날 소프트웨어에서 실행됩니다. 이 행성에는 소프트웨어의 품질에 영향을받지 않는 인간 삶의 구석이 있습니다. 우리가하는 일을 알아야합니다.

이제는 소프트웨어 작성 방법을 배우십시오.
